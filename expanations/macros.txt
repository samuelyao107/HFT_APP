This C++ code defines a lightweight, custom crash-handling utility. It provides two helper functions, `ASSERT` and `FATAL`, designed to catch critical, unrecoverable errors during program execution and terminate the program immediately.

Here is a breakdown of exactly what each part does:

---

## 1. Header Guards and Includes

```cpp
#pragma once
#include <iostream>
#include <cstring>

```

* **`#pragma once`**: A standard preprocessor directive that ensures this header file is included only once during compilation, preventing duplicate definition errors.
* **`<iostream>`**: Brought in to use `std::cerr` and `std::endl` for printing error messages.
* **`<cstring>`**: Included (though technically not strictly needed for this specific snippet, as `std::string` comes implicitly or should be from `<string>`), likely intended for other string utilities in the larger project.

---

## 2. The `ASSERT` Function

```cpp
inline auto ASSERT(bool condition, const std::string &message) noexcept {
    if (!condition) [[unlikely]]{
        std::cerr << "Assertion failed: " << message << std::endl;
        std::abort();
    }
}

```

This function checks if a specific condition is true. If it isn't, it halts the program.

* **`inline auto`**: `inline` suggests to the compiler to embed the function call directly at the call site to eliminate function call overhead. `auto` lets the compiler deduce the return type (which is `void` here).
* **`noexcept`**: Promises the compiler that this function will never throw a C++ exception, allowing for better optimization.
* **`[[unlikely]]` (C++20)**: This is a performance optimization attribute. It tells the compiler that the `if` branch is very rarely triggered. The compiler optimizes the CPU instruction pipeline assuming the condition *will* be true, making the "happy path" faster.
* **`std::cerr`**: Prints the error message to the standard error stream, which is unbuffered (meaning it prints to the screen immediately).
* **`std::abort()`**: Immediately terminates the program abnormally. It does *not* call destructors for local objects, meaning the program stops dead in its tracks right where the failure happened.

---

## 3. The `FATAL` Function

```cpp
inline auto FATAL(const std::string &message) noexcept {
    std::cerr << "Fatal error: " << message << std::endl;
    std::abort();
}

```

This is a simpler tool used when a situation is *already* known to be completely unrecoverable, without needing to check a condition first.

* It immediately prints the "Fatal error" message to `std::cerr`.
* It calls `std::abort()` to crash the program instantly.

---

## Summary of Usage

You would use these functions in a codebase like this:

```cpp
void processAge(int age) {
    // Ensure the input makes sense. If age < 0, the program crashes immediately.
    ASSERT(age >= 0, "Age cannot be negative!"); 
    
    if (age > 150) {
        // We know this is impossible/corrupt data, crash immediately.
        FATAL("Age is impossibly high, memory might be corrupted."); 
    }
}

```

### Key Difference from Standard `assert()`

Unlike the standard C++ `<cassert>` macro (which completely disappears in Release/Production builds), this custom `ASSERT` **will always run**, even in production, because it is written as a standard function.
